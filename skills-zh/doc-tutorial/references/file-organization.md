# 文件组织指南

## 核心原则
关注点分离，保持教程可维护且清晰。

## 推荐结构

```
教程名称/
├── index.md              # 主要教程入口
├── steps/                # 单独的步骤文件（可选，用于大型教程）
│   ├── step-01.md
│   ├── step-02.md
│   └── ...
├── diagrams/             # Mermaid图表（如果从markdown中提取）
│   ├── workflow.mmd
│   ├── sequence.mmd
│   └── ...
├── assets/               # 图像、截图、图标
│   ├── screenshots/
│   │   ├── step-01.png
│   │   └── ...
│   └── icons/
├── scripts/              # 教程特定的脚本
│   ├── setup.sh
│   ├── validate.sh
│   └── ...
└── references/           # 参考材料、速查表等
    ├── glossary.md
    └── faq.md
```

## 何时使用此结构

- **简单教程**（< 5步骤）：将所有内容保存在单个 `index.md` 中
- **中等教程**（5-15步骤）：考虑将步骤分割到 `steps/` 目录中
- **复杂教程**（> 15步骤或大量资产）：使用上面的完整结构

## 命名约定

- 使用小写字母、数字和连字符
- 具有描述性但要简洁
- 对于步骤：`step-01.md`、`step-02.md`等.（使用前导零以便正确排序）
- 对于图表：描述目的，例如 `user-workflow.mmd`、`api-sequence.mmd`

## 资产管理

- 将截图存储在 `assets/screenshots/` 中，文件名包含步骤编号
- 为网络优化图像（适当大小、压缩）
- 使用一致的图像格式（PNG用于截图，SVG用于图标/图表）
- 从markdown文件相对引用资产

## 跨项目兼容性

为了使教程能够适应不同的项目：

1. **使用占位符**：将特定于项目的名称替换为占位符，如 `[your-project-name]`
2. **记录假设**：明确说明教程对项目结构的假设
3. **提供替代方案**：尽可能地展示不同项目类型的变体
4. **使用相对路径**：避免绝对路径；使用相对于项目根目录的路径
5. **版本考虑**：注明教程适用于哪些工具/框架的版本

## 示例：适配教程

原始步骤：
```
1. 导航到 src/components/user-profile/
2. 编辑 UserProfile.js
```

可适配版本：
```
1. 导航到您的组件目录（通常是 src/components/ 或类似位置）
2. 定位用户资料组件文件（可能命名为 UserProfile.js、UserProfile.tsx 等）
3. 编辑用户资料组件文件
```

## 维护提示

- 定期检查文件路径是否仍然存在
- 当UI发生变化时更新资产
- 验证脚本是否仍然适用于当前的工具版本
- 保持图表与描述的流程同步
