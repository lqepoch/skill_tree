# GitHub Issue 工作流

## 目标

使用这个 skill，把 GitHub issue 作为工程工作的系统事实源。issue 应当说明问题本身、可能的来源、涉及的文件或模块、推荐修复方案、验证标准，以及必要的回滚或缓解方案。

## 核心原则

- 将 issue 视为唯一的工作事实源。
- 除非用户明确要求只做探索性分析，否则在创建或更新 issue 之前不要开始改代码。
- 书写 issue 时，要让其他工程师无需额外上下文也能继续推进。
- 将事实、假设和修复建议分开写清楚。
- 优先使用可复用的标签和模板，不要随意发明项目专属的临时命名。
- 在修复被验证并且关联 PR 已合并或已处理之前，不要过早关闭 issue。

## 必填内容

每个 issue 都应该包含：

- 摘要和范围
- 触发条件、环境或复现路径
- 预期行为
- 实际行为或故障表现
- 影响范围和严重度
- 推测的根因或故障来源
- 相关文件、模块、服务、配置或迁移
- 备选修复方案和推荐方案
- 验证计划
- 回滚、缓解或后续跟进说明
- 验收标准

对于 bug，必须补全证据链。对于 feature，要补业务目标和非目标。对于 task，要写清楚可交付结果以及可能变化的文件或系统。

## 工作流

1. 先做分诊。
   - 记录问题、范围、受影响区域、预期结果、业务影响和验收标准。
   - 如果关键细节缺失，就先创建一个分诊 issue，并标记为 `status: needs-info`。
   - 用 issue 正文或后续评论记录未知信息，不要靠猜。
2. 创建或更新 issue。
   - 使用 GitHub MCP 的 `issue_write` 创建或更新 issue。
   - 标题优先采用 `[type] 简短摘要` 的格式。
   - 使用 `assets/github-issue-templates/` 中匹配的模板。
   - 如果模板不够装下问题，就在正文中补齐细节，不要把问题描述压缩得失真。
   - 对 bug issue，必须显式列出：
     - 推测的根因
     - 相关文件或模块
     - 候选修复路径
     - 验证方式
     - 回滚或缓解方案
   - 对 feature issue，必须显式列出：
     - 业务目标
     - 约束和依赖
     - 受影响的文件或服务
     - 发布或迁移考虑
   - 对 task issue，必须显式列出：
     - 可交付物
     - 范围边界
     - 可能涉及的文件或系统
     - 验证清单
3. 应用标签。
   - 至少添加一个 `type:*` 标签和一个 `status:*` 标签。
   - 在已知紧急程度时添加 `priority:*`。
   - 在受影响子系统明确时添加 `area:*`。
   - 保持标签集合小、稳定、可跨仓库复用。
   - 除非仓库本身已经在用，不要随意扩展成一套庞大的项目专属标签体系。
4. 记录进度。
   - 在关键节点都加一条简短评论：分诊完成、计划确认、开始实现、验证完成、准备关闭。
   - 每条更新都要回答：
     - 改了什么
     - 还剩什么
     - 是否存在阻塞
   - 如果被阻塞，要明确写出阻塞原因，并同步更新 issue 状态。
5. 关联修复。
   - PR 准备好时，在 PR 描述中写入 `Closes #123` 或 `Fixes owner/repo#123`。
   - 尽量通过 PR 关联自动关闭 issue，而不是手工关闭。
   - 如果一个修复覆盖多个 issue，要逐个写清楚。
6. 收尾。
   - 添加最终总结评论，说明交付结果、验证结果，以及是否还有后续工作。
   - 只有在验证完成且修复具备稳定性后，才关闭 issue。

## 评论格式

使用统一的进度更新格式，方便快速扫描：

- `Checkpoint`：分诊、计划、实现、验证、完成
- `Change`：已完成的内容
- `Remaining`：还需要继续处理的内容
- `Blocker`：只有在存在阻塞时才写
- `Evidence`：日志、截图、链接、测试结果或复现记录

评论要客观、简短。如果根因还在调查中，要明确说明，并列出当前假设。

## 标签集

优先使用一组小而稳定的核心标签。

| 标签 | 颜色 | 用途 |
| --- | --- | --- |
| `type: bug` | `d73a4a` | 缺陷或回归 |
| `type: feature` | `a2eeef` | 新能力 |
| `type: task` | `cfd3d7` | 任务、清理或后续事项 |
| `type: docs` | `0075ca` | 仅文档工作 |
| `status: needs-triage` | `fbca04` | 需要初步分诊 |
| `status: needs-info` | `7057ff` | 缺少信息，阻碍推进 |
| `status: in-progress` | `5319e7` | 正在处理 |
| `status: blocked` | `d93f0b` | 被依赖、决策或外部问题阻塞 |
| `priority: high` | `b60205` | 紧急工作 |
| `priority: medium` | `fbca04` | 普通优先级 |
| `priority: low` | `0e8a16` | 低优先级 |
| `help wanted` | `008672` | 适合协作 |
| `good first issue` | `7057ff` | 入门型任务 |

- 只使用 6 位十六进制颜色。
- 标签说明保持简短、稳定。
- 除非明确可复用，或者仓库已在使用，否则不要引入项目专属标签。

## 模板指引

优先使用 `.github/ISSUE_TEMPLATE/` 下的 Markdown issue 模板，保证兼容性。

保留一组小而稳定的模板：

- `01-bug-report.md`
- `02-feature-request.md`
- `03-task.md`

让不同仓库的字段尽量对齐，这样同一种 issue 形状可以在不同代码库复用。

对 bug report，模板应要求填写：

- 摘要
- 上下文和环境
- 预期与实际行为
- 复现步骤
- 证据
- 推测根因
- 相关文件
- 修复方案
- 推荐修复
- 验证计划
- 回滚或缓解
- 验收标准

对 feature request，模板应要求填写：

- 摘要
- 业务目标或问题
- 希望达成的结果
- 约束和依赖
- 受影响的文件或系统
- 发布或迁移说明
- 备选方案
- 验收标准

对 task，模板应要求填写：

- 摘要
- 背景
- 范围
- 可能涉及的文件或系统
- 执行计划
- 验证步骤
- 后续或回滚说明
- 验收标准

`config.yml` 只用于让 issue 选择器保持简洁，避免界面杂乱。
