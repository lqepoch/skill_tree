# GitHub Issue 工作流

## 目标

使用这个 skill，把 GitHub issue 作为工程工作的事实源。issue 应该说明问题本身、可能的根因、涉及的文件或模块、推荐修复、验证标准，以及必要的回滚或缓解方案。

## 核心原则

- 把 issue 视为唯一的工作事实源。
- 除非用户明确要求只做探索性分析，否则不要在没有 issue 之前直接开始改代码。
- issue 要写到别人接手后可以继续推进，不依赖隐藏上下文。
- 事实、假设和建议要分开写清楚。
- 优先使用可复用的标签和模板，不要随手发明项目专属命名。
- 只有在修复已验证并且关联 PR 已合并或已解决后，才关闭 issue。

## 必填内容

每个 issue 都应该包含：

- 摘要和范围
- 触发条件、环境或复现路径
- 期望行为
- 实际行为或失败表现
- 影响范围和严重程度
- 推测的根因或故障来源
- 相关文件、模块、服务、配置或迁移
- 候选修复方案和推荐方案
- 验证计划
- 回滚、缓解或后续说明
- 验收标准

对于 bug，要补足证据链。对于 feature，要补业务目标和非目标。对于 task，要写清楚可交付结果，以及可能变化的文件或系统。

## 工作流程

1. 先 triage。
   - 记录问题、范围、受影响区域、预期结果、业务影响和验收标准。
   - 如果关键信息缺失，先创建 triage issue，并标记为 `status: needs-info`。
   - 不要用猜测代替未知信息。
2. 创建或更新 issue。
   - 使用 GitHub issue 模板。
   - 标题优先使用 `[type] 简短摘要`。
   - 使用 `references/issue-templates/` 中对应的模板。
   - 如果模板不够装下问题，就在正文里补充具体例子、证据和假设，不要压缩问题本身。
   - 对于 bug issue，显式列出：
     - 推测根因
     - 相关文件或模块
     - 候选修复路径
     - 验证方式
     - 回滚或缓解方案
   - 对于 feature issue，显式列出：
     - 业务目标
     - 约束和依赖
     - 受影响的文件或服务
     - 发布或迁移考虑
   - 对于 task issue，显式列出：
     - 可交付物
     - 范围边界
     - 可能涉及的文件或系统
     - 验证清单
3. 应用标签。
   - 至少添加一个 `type:*` 标签和一个 `status:*` 标签。
   - 在已知紧急程度时添加 `priority:*`。
   - 在受影响子系统明确时添加 `area:*`。
   - 保持标签集合小、稳定、可跨仓库复用。
   - 除非仓库已经在用，否则不要扩展成庞大的项目专属标签体系。
4. 及时记录进度。
   - 在每个关键检查点都补一条简短评论：triage 完成、方案确认、开始实现、验证完成、准备关闭。
   - 不要等到最后一次性回填。
   - 每次更新都要回答：
     - 改了什么
     - 还剩什么
     - 是否存在阻塞
   - 如果被阻塞，要明确写出阻塞原因，并同步更新 issue 状态。
5. 关联修复。
   - PR 准备好时，必须在 PR 正文写入会自动关闭 issue 的引用，例如 `Closes #123` 或 `Fixes owner/repo#123`。
   - 任何打算关闭工作的 PR 都必须显式包含相关 issue 引用，方便 GitHub 自动关闭。
   - 如果一个修复覆盖多个 issue，要逐个列出，保持引用清晰且不歧义。
6. 收尾。
   - 添加最终总结评论，说明交付结果、验证结果，以及是否还有后续工作。
   - 只有在验证完成且修复足够稳定后，才关闭 issue。

## 评论格式

使用统一的进度更新格式，便于快速扫描：

- `Checkpoint`：triage、plan、implementation、validation、done
- `Change`：本次完成了什么
- `Remaining`：还需要什么
- `Blocker`：只有在存在阻塞时才写
- `Evidence`：日志、截图、链接、测试结果或复现记录

建议在这些时点更新：

- triage 完成
- 方案确认
- 开始实现
- 验证完成
- 准备关闭

## 标签集

优先使用一组小而稳定的核心标签。

| 标签 | 颜色 | 用途 |
| --- | --- | --- |
| `type: bug` | `d73a4a` | 缺陷或回归 |
| `type: feature` | `a2eeef` | 新能力 |
| `type: task` | `cfd3d7` | 任务、清理或后续事项 |
| `type: docs` | `0075ca` | 纯文档工作 |
| `status: needs-triage` | `fbca04` | 需要初步分流 |
| `status: needs-info` | `7057ff` | 信息不足，阻塞推进 |
| `status: in-progress` | `5319e7` | 正在处理 |
| `status: blocked` | `d93f0b` | 被依赖、决策或外部问题阻塞 |
| `priority: high` | `b60205` | 紧急工作 |
| `priority: medium` | `fbca04` | 普通优先级 |
| `priority: low` | `0e8a16` | 低优先级 |
| `help wanted` | `008672` | 适合协作 |
| `good first issue` | `7057ff` | 入门任务 |

- 只使用 6 位十六进制颜色。
- 标签说明保持简短且一致。
- 除非明确可复用，或者仓库已经在使用，否则不要引入项目专属标签。

## 模板指引

优先使用 `references/issue-templates/` 下的 Markdown issue 参考模板，保证兼容性和可复用性。

保留一组小而稳定的模板：

- `01-bug-report.md`
- `02-feature-request.md`
- `03-task.md`

让不同仓库的字段尽量对齐，这样同一类 issue 可以在不同代码库里复用。

对于 bug report，模板应要求填写：

- 摘要
- 上下文和环境
- 期望与实际行为
- 复现步骤
- 证据
- 推测根因
- 相关文件
- 修复方案
- 推荐修复
- 验证计划
- 回滚或缓解
- 验收标准

对于 feature request，模板应要求填写：

- 摘要
- 业务目标或问题
- 期望结果
- 约束和依赖
- 受影响的文件或系统
- 发布或迁移说明
- 备选方案
- 验收标准

对于 task，模板应要求填写：

- 摘要
- 背景
- 范围
- 可能涉及的文件或系统
- 执行计划
- 验证步骤
- 后续或回滚说明
- 验收标准

`config.yml` 只用于让 issue 选择器保持简洁，避免界面杂乱。
