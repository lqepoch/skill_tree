# issue 指南

当仓库需要遵循 issue 优先的工作流时，使用这个 skill。

## issue 结构

每个高质量 issue 都应该说明：

- 问题、缺口或需求是什么
- 发生在哪儿
- 如何复现或证明
- 预期结果是什么
- 为什么重要
- 相关文件、模块、服务、配置或迁移是什么
- 推荐的修复或下一步是什么
- 如何验证成功
- 是否需要回滚或缓解

如果根因尚未确认，就把它标记为假设，并写出支撑这个假设的证据。

## Bug 标准

bug issue 应该记录：

- 观察到的行为
- 预期行为
- 环境信息
- 复现步骤
- 日志、链路、截图或链接
- 推测的故障来源
- 相关文件以及它们为什么可疑
- 候选修复方案，以及为什么优先某个方案
- 验证和回滚计划

如果存在多个可能的故障来源，要按优先级列出，并说明每个来源是如何被支持或排除的。

## Feature 标准

feature issue 应该记录：

- 业务目标或用户需求
- 当前行为为什么不够用
- 范围和非目标
- 约束和依赖
- 受影响的文件或系统
- 发布、迁移或兼容性考虑
- 备选方案
- 验收标准

## Task 标准

task issue 应该记录：

- 具体可交付结果
- 任务存在的原因
- 范围边界
- 可能会修改的文件或系统
- 执行计划
- 验证步骤
- 后续清理或收尾事项

## 模板规则

- 优先使用 Markdown issue 模板，兼容性最好。
- 保持字段在不同仓库中稳定一致。
- 维持一组小而稳定的模板集合：
  - `01-bug-report.md`
  - `02-feature-request.md`
  - `03-task.md`
- 提供一个简单的 `config.yml`，让 issue 选择器保持干净。

## 标签规则

- 保持核心标签体系尽量小。
- 尽可能在不同仓库中复用相同的标签名。
- 推荐标签族：
  - `type:*`
  - `status:*`
  - `priority:*`
  - `area:*`
- 标签颜色要清晰、易读、可区分。
- 只使用 6 位十六进制颜色。

## 评论节奏

当工作跨过清晰的阶段时，补一条进度评论：

- 分诊完成
- 计划确认
- 开始实现
- 验证完成
- 准备关闭

每条评论都要回答：

- 改了什么
- 还剩什么
- 是否有阻塞

如果根因仍在调查中，要写出当前主要假设，以及下一步准备做的检查。

## PR 关联

当变更准备好时，在 PR 描述里写入 `Closes #<issue>` 或 `Fixes owner/repo#<issue>`。
尽量使用这种关联方式，而不要手工关闭 issue。
如果一个变更会关闭多个 issue，要把每个 issue 都明确列出来。
