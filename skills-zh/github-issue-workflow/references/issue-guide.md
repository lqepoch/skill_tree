# issue 指南

当仓库需要 issue 优先的工作流时，使用这个 skill。

## 模板规则

- 优先使用 Markdown issue 模板，兼容性最好。
- 模板字段在各仓库中保持稳定。
- 维持一个小而稳定的模板集合：
  - `01-bug-report.md`
  - `02-feature-request.md`
  - `03-task.md`
- 提供一个简单的 `config.yml`，让 issue 选择器保持清晰。

## 标签规则

- 保持核心标签体系尽量小。
- 尽量在不同仓库中复用相同的标签名。
- 推荐标签族：
  - `type:*`
  - `status:*`
  - `priority:*`
  - `area:*`
- 标签颜色要清晰、易读、可区分。
- 只使用 6 位十六进制颜色。

## 评论节奏

在这些关键节点补充进度评论：

- 分诊完成
- 计划确认
- 开始实现
- 验证完成
- 准备关闭

每条评论都回答三个问题：

- 改了什么
- 还剩什么
- 是否有阻塞

## PR 关联

准备合并时，在 PR 描述里写入 `Closes #<issue>` 或 `Fixes owner/repo#<issue>`。
能自动关闭时，优先使用这种关联方式，而不是手工关闭。
