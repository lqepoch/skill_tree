# 缺陷报告

> 使用说明：这个模板的目标不是“快速填个标题”，而是让后续接手的人可以直接开始定位、复现、修复和验证。
>
> 如果你暂时不知道根因，也不要空着。请明确写出“当前假设”以及支持该假设的证据。

- 问题一句话说明：
- 影响范围一句话说明：
- 你希望最终达到的状态：

> 在提交订单时，支付确认页偶发返回 500，导致用户无法完成付款。

- 浏览器 / 操作系统 / 运行时 / 版本：
- 是否只在特定账号、租户、数据集或地域出现：

- 页面：`/checkout/confirm`
- 版本：`web-2026.03.18`
- 最近变更：3 月 21 日上线了支付网关超时重试逻辑

## 13. 接受标准

把“完成”的定义写成可检查的清单。

- [ ] 相关 issue 可以进入关闭状态

## 14. 推荐评论格式

在 issue 评论里更新进度时，建议固定成下面的结构：

- `Checkpoint`：triage / plan / implementation / validation / done
- `Change`：本次完成了什么
- `Remaining`：还剩什么
- `Blocker`：当前阻塞点，只有真的被卡住时才写
- `Evidence`：日志、截图、链接、测试结果或复现记录

### 示例评论

```text
Checkpoint: plan
Change: 已确认失败集中在支付确认链路，定位到重试逻辑和超时配置
Remaining: 需要验证是否为上游网关抖动还是本地连接未释放
Blocker: 暂无
Evidence: request_id=...
```

## 15. 示例成品

### 示例标题

`[bug] Checkout payment confirmation returns 500 after retry`

### 示例正文

- 摘要：结账确认页在支付重试后偶发 500
- 环境：生产，`web-2026.03.18`
- 期望：用户支付成功后立即看到成功页
- 实际：连续点击后出现 500 页面
- 复现步骤：进入购物车，选择信用卡，连续点击确认支付
- 证据：`request_id=...`，错误日志显示 `PaymentService timeout`
- 根因假设：同步重试阻塞连接池
- 相关文件：`src/payments/confirm.ts`、`src/lib/http/retry.ts`
- 修复思路：改为异步重试并降低同步链路超时敏感度
- 验证计划：单测 + 预发重复支付验证
- 接受标准：连续 10 次执行无 500 且订单均成功
